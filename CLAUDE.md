# Transfer In Plan Management System

## What This Application Is

An ASP.NET Core 8.0 MVC web application for **retail supply chain planning**. It calculates how much stock to transfer into stores (Transfer In Plan) and how much to purchase at DC/warehouse level (Purchase Plan), broken down by store, category, and week. It also generates sub-level plans by splitting these calculations across 4 dimensions (Macro MVGR, Size, Segment, Vendor) using contribution percentages.

**URL**: `localhost:5005`  
**Database**: SQL Server — database name `planning`  
**Connection string key**: `PlanningDatabase` in `appsettings.json`

## Stored Procedures

| SP | Purpose | Called From |
|----|---------|------------|
| `SP_GENERATE_TRF_IN_PLAN` | Transfer In Plan for specific store/week range | PlanController.Execute (via PlanService) |
| `SP_GENERATE_PURCHASE_PLAN` | Purchase Plan for week range + optional RDC/MajCat | PurchasePlanController.Execute |
| `SP_RUN_ALL_PLANS` | Full run: truncate + TRF + PP for all stores (bulk V3) | PlanJobService (background) |
| `SP_GENERATE_SUB_LEVEL_TRF` | Sub-level TRF per level with level-specific stock | SubLevelController.RunGenerate / SubLevelJobService |
| `SP_GENERATE_SUB_LEVEL_PP` | Sub-level PP per level reading from SUB_LEVEL_TRF_PLAN | SubLevelController.RunGenerate / SubLevelJobService |

### TRF Algorithm (week chaining)
```
For each ST_CD + MAJ_CAT + WEEK (or + SUB_VALUE for sub-level):
  Week 1: OP_STK = Store Stock
  Shrinkage = OP_STK × 8% × (SSN in S,PS ? 1.0 : 0.5)
  NET_CF = OP_STK - Shrinkage
  MBQ = Display + Next Week Sale
  TRF_IN = MAX(0, MBQ + Sale - NET_CF)  // rounded to BIN_CAP multiples in main SP
  CL_STK = NET_CF + TRF_IN (or variations based on MBQ vs NET_CF)
  Week 2+: OP_STK = previous week CL_STK → chain continues
```

### PP Algorithm (DC chaining)
```
For each RDC_CD + MAJ_CAT + WEEK (or + SUB_VALUE):
  Aggregates TRF_OUT from TRF_IN_PLAN (SUM of TRF_IN_STK_Q by RDC)
  Week 1: DC_STK = DC Stock table
  BGT_DC_MBQ_SALE = sum of next 4 weeks sale
  BGT_DC_CL_MBQ = MIN(sum of next 4 weeks TRF_OUT, MBQ_SALE)
  BGT_PUR_Q = MAX(0, TTL_TRF_OUT + CL_MBQ - CF_STK)
  BGT_DC_CL_STK = CF_STK + PUR_Q - TTL_TRF_OUT
  Week 2+: DC_STK = previous week CL_STK → chain continues
```

### Sub-Level Difference
Sub-level runs the **exact same algorithm** but with:
- Sale Qty × CONT_PCT from contribution table
- Display Qty × CONT_PCT from contribution table
- Store stock from `SUB_ST_STK_{level}` instead of `STORE_STOCK`
- DC stock from `SUB_DC_STK_{level}` instead of `DC_STOCK`
- Granularity: ST_CD + MAJ_CAT + **SUB_VALUE** + WEEK

## Background Job Services (Singletons)

### PlanJobService
- Runs `SP_RUN_ALL_PLANS` in background (fire-and-forget Task.Run)
- Phases: Initializing → Cleaning (truncate) → Running (SP) → Done/Error
- Polled via `PlanController.JobStatus()` every 5s
- Thread-safe with lock object

### SubLevelJobService
- Runs `SP_GENERATE_SUB_LEVEL_TRF` + `SP_GENERATE_SUB_LEVEL_PP` per level sequentially
- Tracks: Phase (TRF/PP), CurrentLevel, LevelsCompleted/TotalLevels
- Polled via `SubLevelController.SubJobStatus()` every 5s
- Same singleton pattern as PlanJobService

## Key Patterns & Conventions

1. **Column naming**: SQL uses `[ST-CD]`, `[MAJ-CAT]`, `[BIN CAP]` (with spaces/hyphens). EF models use `[Column("ST-CD")]` attribute mapping.
2. **Nullable types**: Most decimal columns are `decimal?` in models. Always use `?? 0` when summing.
3. **Raw SQL**: KPI queries and dashboard use raw `SqlConnection` + `SqlCommand` for performance. CRUD uses EF Core.
4. **NOLOCK**: All read queries use `WITH (NOLOCK)` hint.
5. **Week columns**: Sale/Display tables have 48 weekly columns (`WK-1` to `WK-48`). SPs use UNPIVOT to convert to rows.
6. **TempData**: Success/Error messages passed via `TempData["SuccessMessage"]` / `TempData["ErrorMessage"]`. Layout renders them globally.
7. **ViewBag**: Filter dropdowns loaded into ViewBag (lists of strings or model objects).
8. **Streaming CSV**: Large exports (13M+ rows) stream directly to `Response.Body` using `StreamWriter`.
9. **DBNull handling**: When passing NULL to `ExecuteSqlRawAsync`, use typed `SqlParameter` with `SqlDbType.NVarChar` — do NOT pass `DBNull.Value` directly (causes "store type mapping for DBNull" error).
10. **Temp tables in dynamic SQL**: `SELECT INTO #TempTable` inside `sp_executesql` creates the temp table in child scope — it's dropped when sp_executesql returns. Fix: `CREATE TABLE #TempTable` in parent scope, then `INSERT INTO` from dynamic SQL.
11. **UNPIVOT alias**: After `FROM table UNPIVOT (...) u`, the original table alias is no longer valid. Use `u.column` not `original.column`.

## Common Issues & Fixes

| Issue | Cause | Fix |
|-------|-------|-----|
| "store type mapping for DBNull" | Passing `DBNull.Value` to `ExecuteSqlRawAsync` | Use typed `SqlParameter` with explicit `SqlDbType` |
| "Cannot find object #SC" | Temp table created inside `sp_executesql` | Create table in parent scope, INSERT from dynamic SQL |
| "multi-part identifier sq.ST-CD could not be bound" | Using original alias after UNPIVOT | Use the UNPIVOT alias `u` instead |
| Build error MSB3027 "file locked" | App is running, exe locked | Stop the app before rebuilding |
| DataTables "Incorrect column count" | Empty table body | Guard: check `tbody tr` count before DataTable init |
| Pivot CSV download fails "Check internet connection" | NULL `FY_WEEK` crashes `GetInt32()` | Add `ISNULL()` wrappers + `FY_WEEK IS NOT NULL` filter |
| Duplicate TempData error alerts | Both Layout and View read TempData | Remove TempData display from individual views (Layout handles it) |
